Software Requirements Specification (SRS)

No.: Requirement: Description:
1 Introduction The Requirements Manager is a desktop software which allows its user to
  • Organize the requirements of (software) projects in a tree structure
  • Describe the requirements in rich text format
  • Document, which classes and functions implement a requirement
  • Assess the risk of requirements and of their implementation
  • Document, which tests are in place to mitigate a given risk and to verify that a given requirement is fullfilled
Class and function declarations are extracted automatically from the source code along with their comments and can be mapped to the requirements they implement.

The software is written in C++ and based on the Qt library. Therefore it can easily be compiled for use on Linux, Windows and Mac.
2 Multi Platform GUI program The software should run as a desktop application on Windows, Linux and Mac OS and provide a graphical user interface.
3 Hierarchical structure Requirements are stored in a tree structure of arbitrary depth. Each tree node consists of a title and a description.
4 The structure is editable The requirement tree structure can be edited by the user.
5 Requirement types One of three different "types" can be assigned to each requirement. The type of each requirement is made visible in the requirement tree view. This provides the possibility to structure the requirements tree into three layers of depth:
  • User Requirements
  • Functional Requirements
  • Design Requirements
6 Rich text descriptions For each requirement a rich text description can be edited to specify its details.
7 Attributes For each project an arbitrary number of user-configurable "requirement attributes" can be used. Attributes store additional information about requirements e.g. their state, a time estimation, a comment ... Attributes have the following properties
  • name
  • data type
  • value
  • boolean
  • text
  • ranking (1 to 5)
  • integer
8 Links For each project an arbitrary number of user-configurable "requirement links" can be used. Links can be used to specify relationships like "requirement A affects / contradicts / ... requirements B, C, D, ..."
9 Risk Assessment For each requirement an assessment of their risks can be edited:
  • Several "risk scenarios" can be added to each requirement
  • Several "preventive actions" can be assigned to each risk scenario
  • Each risk risk scenario has a final risk resulting from the sum of preventive actions
10 Source Code Extraction
  • When used with software projects the Requirements Manager can extract declarations of classes, functions and automated tests along with their comments from the project's source files.
  • Classes and functions can be added as Source Code Documentation to the Design Requirements
  • Automated tests can be added to risk assessments
11 Saving and Loading of Projects All information edited in the software can be saved to file and loaded again.
12 Exporting Reports The information stored in a project file can be exported to reports. Possible formats are
  • HTML
  • PDF
  • OpenDocument

Functional Requirements Specification (FRS)

No.: SRS.FRS: Description:
1 Introduction The Requirements Manager is a desktop software which allows its user to
  • Organize the requirements of (software) projects in a tree structure
  • Describe the requirements in rich text format
  • Document, which classes and functions implement a requirement
  • Assess the risk of requirements and of their implementation
  • Document, which tests are in place to mitigate a given risk and to verify that a given requirement is fullfilled
Class and function declarations are extracted automatically from the source code along with their comments and can be mapped to the requirements they implement.

The software is written in C++ and based on the Qt library. Therefore it can easily be compiled for use on Linux, Windows and Mac.
2 Multi Platform GUI program The software should run as a desktop application on Windows, Linux and Mac OS and provide a graphical user interface.
2.1 C++ and Qt Library The software is implemented in the C++ programming language and makes use of the Qt framework.
3 Hierarchical structure Requirements are stored in a tree structure of arbitrary depth. Each tree node consists of a title and a description.
3.1 Display to user
Requirements are displayed in tree form similar to the display of a file system tree.
4 The structure is editable The requirement tree structure can be edited by the user.
4.1 Edit titles The requirements' titles can be edited by double-clicking on a requirement or by selecting the requirement and pressing F2.
4.2 Adding as child A requirement can be added as child of another requirement. Triggers are
  • a tool button
  • the keyboard shortcut ctrl-shift-r
4.3 Adding as sibling A requirement can be added as a sibling of another requirement. Triggers are
  • a tool button
  • the keyboard shortcut ctrl-r
4.4 Removing The currently selected requirement can be removed. Triggers are
  • a tool button
  • the keyboard shortcut delete
4.4.1 Saveguard question The user is asked if he 'really wants to delete the requirement and all its children'.
4.5 Moving Requirements can be moved by drag and drop within the tree view.
4.6 Copying Requirements can be copied within the tree view by drag and drop with the keyboard modifier 'ctrl'.
5 Requirement types One of three different "types" can be assigned to each requirement. The type of each requirement is made visible in the requirement tree view. This provides the possibility to structure the requirements tree into three layers of depth:
  • User Requirements
  • Functional Requirements
  • Design Requirements
5.1 Display The requirement types are displayed color-coded in the requirements tree view: Each requirement's title is displayed in a font color corresponding to its type:
  • User Requirement: black
  • Functional Requirement: blue
  • Design Requirement: magenta
5.2 Editing The requirement type can be selected in a context menu of the requirements tree view. A right click on a requirement opens a context menu which lets the user choose one of the three types.
6 Rich text descriptions For each requirement a rich text description can be edited to specify its details.
6.1 Bold Text Text can be formatted bold. Triggers are
  • A tool button
  • The keyboard shortcut ctrl-b
6.2 Lists Text can be formatted in lists. Triggers are
  • A tool button
  • The keyboard shortcut ctrl-l
6.3 Italic Text Text can be formatted italic. Triggers are
  • A tool button
  • The keyboard shortcut ctrl-i
6.4 Code Text can be formatted in code form. Triggers are
  • A tool button
  • A the keyboard
  • ctrl-d
6.5 Images Images can be inserted into the rich text description.
7 Attributes For each project an arbitrary number of user-configurable "requirement attributes" can be used. Attributes store additional information about requirements e.g. their state, a time estimation, a comment ... Attributes have the following properties
  • name
  • data type
  • value
  • boolean
  • text
  • ranking (1 to 5)
  • integer
7.1 Attribute Context Editor
7.2 Display of Attributes
7.3 Editing Attributes
8 Links For each project an arbitrary number of user-configurable "requirement links" can be used. Links can be used to specify relationships like "requirement A affects / contradicts / ... requirements B, C, D, ..."
8.1 Configuration Dialog
  • Via the menu bar entry "Edit" a configuration dialog for the available links can be opened.
  • The dialog shows the list of defined link types (e.g. "affects", "contradicts", etc).
  • Link types can be added, renamed and deleted.
8.2 Display
  • The links for the selected requirement are shown in a tree view at the right border of the main window.
  • The uppermost level of the tree structure contains the link types. The link types' children are the actual links of the selected requirment.
8.3 Adding Links To add links, the user switches the application into linking mode by activating a toggle button. In linking mode, double click on a requirement entry adds the requirement to the currently selected link type.
8.4 Removing links Links can be removed by selecting them and either clicking a remove button (minus sign) or pressing delete.
8.5 Saving Links The information about links is saved into the project file.
8.6 Loading Links The information about links can be loaded to the project file.
9 Risk Assessment For each requirement an assessment of their risks can be edited:
  • Several "risk scenarios" can be added to each requirement
  • Several "preventive actions" can be assigned to each risk scenario
  • Each risk risk scenario has a final risk resulting from the sum of preventive actions
9.1 Display When a requirement in the tree view is selected, the table view "Risks" is updated with the list of risks related to this requirement. Each entry in the "Risks" view consists of the following fields Title | Initial risk (text (high, acceptable, low) and color coded (red, yellow, green)) | Final risk When a risk in the list is selected, a detailed report of the risk is shown in the "Risk / Action Description" text view.

10 Source Code Extraction
  • When used with software projects the Requirements Manager can extract declarations of classes, functions and automated tests along with their comments from the project's source files.
  • Classes and functions can be added as Source Code Documentation to the Design Requirements
  • Automated tests can be added to risk assessments
10.1 Extract information from code Information about structure and function of the code is extracted from the source code files

10.2 Multiple programming languages Documentation can be extracted from various programming languages

12 Exporting Reports The information stored in a project file can be exported to reports. Possible formats are
  • HTML
  • PDF
  • OpenDocument
