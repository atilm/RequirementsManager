Software Requirements Specification (SRS)

No.: Requirement: Description: regulatory functional
1 Hierarchical structure Requirements are stored in a tree structure of arbitrary depth. The following editing actions are possible
  • adding and removing requirements
  • moving requirements
  • copying requirements
2 The structure is editable The requirement tree structure can be edited by the user.
3 Requirement types One of three different "types" can be assigned to each requirement. The type of each requirement is made visible in the requirement tree view. This provides the possibility to structure the requirements tree into three layers of depth:
  • User Requirements
  • Functional Requirements
  • Design Requirements
4 Rich text descriptions For each requirement a rich text description can be edited to specify its details.
5 Saving and Loading of Projects All information edited in the software can be saved to file and loaded again.
6 Exporting Reports The information stored in a project file can be exported to reports. Possible formats are
  • HTML
  • PDF
  • OpenDocument
7 Digital Signatures Project files and/or exported reports can be digitally signed and signatures can be veryfied using the Requirements Manager or an additional tool.
8 Attributes For each project an arbitrary number of user-configurable "requirement attributes" can be used. Attributes store additional information about requirements e.g. their state, a time estimation, a comment ... Attributes have the following properties
  • name
  • data type
  • value
  • boolean
  • text
  • ranking (1 to 5)
  • integer
9 Links For each project an arbitrary number of user-configurable "requirement links" can be used. Links can be used to specify relationships like "requirement A affects / contradicts / ... requirements B, C, D, ..."
10 Risk Assessment For each requirement an assessment of their risks can be edited:
  • Several "risk scenarios" can be added to each requirement
  • Several "preventive actions" can be assigned to each risk scenario
  • Each risk risk scenario has a final risk resulting from the sum of preventive actions
11 Source Code Extraction
  • When used with software projects the Requirements Manager can extract declarations of classes, functions and automated tests along with their comments from the project's source files.
  • Classes and functions can be added as Source Code Documentation to the Design Requirements
  • Automated tests can be added to risk assessments
12 Full-Text Search The user can perform a full text search in the requirements document.

Functional Requirements Specification (FRS)

No.: SRS.FRS: Description:
1 Hierarchical structure Requirements are stored in a tree structure of arbitrary depth. The following editing actions are possible
  • adding and removing requirements
  • moving requirements
  • copying requirements
1.1 Display to user
  • Requirements are displayed in tree form similar to the display of a file system tree.
  • It is possible to "collapse nodes" to make their sub-requirements invisible
2 The structure is editable The requirement tree structure can be edited by the user.
2.1 Edit titles The requirements' titles can be edited by double-clicking on a requirement or by selecting the requirement and pressing F2.
2.2 Adding as child A requirement can be added as child of another requirement. Triggers are
  • a tool button
  • the keyboard shortcut ctrl-shift-r
2.3 Adding as sibling A requirement can be added as a sibling of another requirement. Triggers are
  • a tool button
  • the keyboard shortcut ctrl-r
2.4 Removing The currently selected requirement can be removed. Triggers are
  • a tool button
  • the keyboard shortcut delete
2.5 Moving Requirements can be moved by drag and drop within the tree view.
2.6 Copying Requirements can be copied within the tree view by drag and drop with the keyboard modifier 'ctrl'.
3 Requirement types One of three different "types" can be assigned to each requirement. The type of each requirement is made visible in the requirement tree view. This provides the possibility to structure the requirements tree into three layers of depth:
  • User Requirements
  • Functional Requirements
  • Design Requirements
3.1 Display The requirement types are displayed color-coded in the requirements tree view: Each requirement's title is displayed in a font color corresponding to its type:
  • User Requirement: black
  • Functional Requirement: blue
  • Design Requirement: magenta
3.2 Editing The requirement type can be selected in a context menu of the requirements tree view. A right click on a requirement opens a context menu which lets the user choose one of the three types.
4 Rich text descriptions For each requirement a rich text description can be edited to specify its details.
4.1 Bold Text Text can be formatted bold. Triggers are
  • A tool button
  • The keyboard shortcut ctrl-b
4.2 Lists Text can be formatted in lists. Triggers are
  • A tool button
  • The keyboard shortcut ctrl-l
4.3 Italic Text Text can be formatted italic. Triggers are
  • A tool button
  • The keyboard shortcut ctrl-i
4.4 Code Text can be formatted in code form. Triggers are
  • A tool button
  • A the keyboard
  • ctrl-d
4.5 Images Images can be inserted into the rich text description.
5 Saving and Loading of Projects All information edited in the software can be saved to file and loaded again.
5.1 Display of unsaved changes
5.2 Load
5.3 Save
5.4 Save As
5.5 Storage format The main storage format for the project data is a single xml file. Data that cannot be stored in the xml file (e.g. images) are stored in a automatically created folder in the same directory as the xml file.
6 Exporting Reports The information stored in a project file can be exported to reports. Possible formats are
  • HTML
  • PDF
  • OpenDocument
7 Digital Signatures Project files and/or exported reports can be digitally signed and signatures can be veryfied using the Requirements Manager or an additional tool.
8 Attributes For each project an arbitrary number of user-configurable "requirement attributes" can be used. Attributes store additional information about requirements e.g. their state, a time estimation, a comment ... Attributes have the following properties
  • name
  • data type
  • value
  • boolean
  • text
  • ranking (1 to 5)
  • integer
8.1 Attribute Context There's one shared instance of class AttributeContext. It store's which attributes exist and passes information about changes to all requirements. The AttributContext can be edited in a dedicated dialog.
8.2 Attribute Container Every requirement contains an object of class AttributeContainer. AttributeContainer stores a pointer to the AttributContext instance and a list of concrete attributes for the parent requirement.
8.3 RequirementToModelMapper The class RequirmentToModelMapper returns attributes for the DisplayRole of Columns greater 0.
8.4 Attribute Context Editor
8.5 Display of Attributes
8.6 Editing Attributes
9 Links For each project an arbitrary number of user-configurable "requirement links" can be used. Links can be used to specify relationships like "requirement A affects / contradicts / ... requirements B, C, D, ..."
9.1 class LinkContext There's one shared instance of class LinkContext. It store's which links exist and passes information about changes to all LinkContainers (one in each requirement). Link Context inherites from QAbstractTableModel. The AttributContext can be edited in a dedicated dialog.
9.2 class LinkContainer
9.3 Configuration Dialog
  • Via the menu bar entry "Edit" a configuration dialog for the available links can be opened.
  • The dialog shows the list of defined link types (e.g. "affects", "contradicts", etc).
  • Link types can be added, renamed and deleted.
9.4 Display
  • The links for the selected requirement are shown in a tree view at the right border of the main window.
  • The uppermost level of the tree structure contains the link types. The link types' children are the actual links of the selected requirment.
9.5 Adding Links To add links, the user switches the application into linking mode by activating a toggle button. In linking mode, double click on a requirement entry adds the requirement to the currently selected link type.
9.6 Removing links Links can be removed by selecting them and either clicking a remove button (minus sign) or pressing delete.
9.7 Saving Links The information about links is saved into the project file.
9.8 Loading Links The information about links can be loaded to the project file.
10 Risk Assessment For each requirement an assessment of their risks can be edited:
  • Several "risk scenarios" can be added to each requirement
  • Several "preventive actions" can be assigned to each risk scenario
  • Each risk risk scenario has a final risk resulting from the sum of preventive actions
10.1 Display When a requirement in the tree view is selected, the table view "Risks" is updated with the list of risks related to this requirement. Each entry in the "Risks" view consists of the following fields Title | Initial risk (text (high, acceptable, low) and color coded (red, yellow, green)) | Final risk When a risk in the list is selected, a detailed report of the risk is shown in the "Risk / Action Description" text view.
10.2 Adding risks
10.3 Assessing risks
10.4 Adding preventive actions
10.5 Assessing preventive actions
11 Source Code Extraction
  • When used with software projects the Requirements Manager can extract declarations of classes, functions and automated tests along with their comments from the project's source files.
  • Classes and functions can be added as Source Code Documentation to the Design Requirements
  • Automated tests can be added to risk assessments
11.1 Extract information from code Information about structure and function of the code is extracted from the source code files
11.2 Multiple programming languages Documentation can be extracted from various programming languages
11.3 Display of structure
11.4 Assignement to requirements
